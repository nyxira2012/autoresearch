# autoresearch

这不仅是一项实验，更旨在让 LLM 进行自主研究。

## 设置 (Setup)

为了建立一项新实验，请与用户协作完成以下步骤：

1. **商定运行标签 (run tag)**：根据今天的日期提议一个标签（例如 `mar5`）。分支 `autoresearch/<tag>` 必须尚不存在 —— 这是一次全新的运行。
2. **创建分支**：从当前的 master 分支执行 `git checkout -b autoresearch/<tag>`。
3. **阅读相关文件**：该仓库非常精简。请阅读以下文件以获取完整上下文：
   - `README.md` —— 仓库背景信息。
   - `prepare.py` —— 固定的常量、数据准备、分词器、数据加载器、评估。请勿修改。
   - `train.py` —— 你修改的文件。包含模型架构、优化器、训练循环。
4. **验证数据是否存在**：检查 `~/.cache/autoresearch/` 是否包含数据分片和分词器。如果没有，请告知人类运行 `uv run prepare.py`。
5. **初始化 results.tsv**：创建仅包含标题行的 `results.tsv`。基准测试结果将在第一次运行后记录。
6. **确认并开始**：确认设置无误。

一旦获得确认，即可启动实验。

## 实验过程 (Experimentation)

每个实验在单个 GPU 上运行。训练脚本运行的时间**固定为 5 分钟**（实际训练时间，不包括启动/编译开销）。你只需通过以下命令启动它：`uv run train.py`。

**你可以做的：**
- 修改 `train.py` —— 这是你编辑的唯一文件。任何内容都可以修改：模型架构、优化器、超参数、训练循环、批次大小、模型大小等。

**你不能做的：**
- 修改 `prepare.py`。它是只读的。它包含了固定的评估方案、数据加载、分词器以及训练常量（时间预算、序列长度等）。
- 安装新包或添加依赖项。你只能使用 `pyproject.toml` 中已有的内容。
- 修改评估体系。`prepare.py` 中的 `evaluate_bpb` 函数是衡量指标的唯一事实来源。

**目标很简单：获得最低的 val_bpb。** 由于时间预算是固定的，你不需要担心训练时间 —— 总是 5 分钟。任何尝试都是允许的：更改架构、优化器、超参数、批次大小、模型大小。唯一的约束是代码能运行而不崩溃，并在时间预算内完成。

**显存 (VRAM)** 是一个软约束。为了获得显著的 val_bpb 提升，适度的显存增加是可以接受的，但不应出现剧烈的膨胀。

**简洁准则 (Simplicity criterion)**：在同等条件下，越简单越好。如果一个微小的改进引入了丑陋的复杂性，那就不值得。反之，如果删除某些内容并获得相同或更好的结果，那就是一个伟大的成果 —— 这是简洁性的胜利。在评估是否保留某项更改时，请权衡复杂性成本与改进幅度。增加 20 行 hack 代码仅换取 0.001 的 val_bpb 提升？可能不值得。通过删除代码获得 0.001 的提升？绝对要保留。代码简单得多但性能基本一致？保留。

**首次运行**：你的第一次运行应始终是为了建立基准，因此请原样运行训练脚本。

## 输出格式 (Output format)

脚本运行结束后，会打印如下摘要：

```
---
val_bpb:          0.997900
training_seconds: 300.1
total_seconds:    325.9
peak_vram_mb:     45060.2
mfu_percent:      39.80
total_tokens_M:   499.6
num_steps:        953
num_params_M:     50.3
depth:            8
```

请注意，脚本配置为始终在 5 分钟后停止，因此根据该计算机的计算平台，数字可能会有所不同。你可以从日志文件中提取关键指标：

```bash
grep "^val_bpb:" run.log
```

## 记录结果 (Logging results)

实验完成后，将其记录到 `results.tsv` 中（制表符分隔，不要用逗号分隔 —— 描述中的逗号会破坏格式）。

TSV 文件包含标题行和 5 列：

```
commit	val_bpb	memory_gb	status	description
```

1. git commit hash (短哈希，7 位)
2. 达到的 val_bpb (例如 1.234567) —— 如果崩溃请使用 0.000000
3. 显存峰值 (GB)，保留一位小数 (例如 12.3 —— 将 peak_vram_mb 除以 1024) —— 如果崩溃请使用 0.0
4. 状态 (status)：`keep` (保留), `discard` (放弃), 或 `crash` (崩溃)
5. 本次实验尝试内容的简短文字描述

示例：

```
commit	val_bpb	memory_gb	status	description
a1b2c3d	0.997900	44.0	keep	baseline
b2c3d4e	0.993200	44.2	keep	increase LR to 0.04
c3d4e5f	1.005000	44.0	discard	switch to GeLU activation
d4e5f6g	0.000000	0.0	crash	double model width (OOM)
```

## 实验循环 (The experiment loop)

实验在专用分支上运行（例如 `autoresearch/mar5` 或 `autoresearch/mar5-gpu0`）。

无限循环：

1. 查看 git 状态：我们当前所在的分支/提交
2. 通过直接修改代码来调整 `train.py` 中的实验想法。
3. git commit
4. 运行实验：`uv run train.py > run.log 2>&1` (重定向所有输出 —— 不要使用 tee 或让输出淹没你的上下文)
5. 读取结果：`grep "^val_bpb:\|^peak_vram_mb:" run.log`
6. 如果 grep 输出为空，则说明运行崩溃。运行 `tail -n 50 run.log` 阅读 Python 堆栈跟踪并尝试修复。如果尝试几次后仍无法运行，请放弃。
7. 在 tsv 中记录结果（注意：不要提交 results.tsv 文件，让它保持为 git 未跟踪状态）
8. 如果 val_bpb 有所改善（变低），则“推进”分支，保留该 git 提交
9. 如果 val_bpb 持平或变差，则 git reset 回到起始位置

核心理念是你作为一个完全自主的研究员在尝试各种新事物。如果有效，保留；如果无效，放弃。你通过推进分支来进行迭代。如果你觉得在某种程度上陷入了困境，可以回溯，但你应该非常非常谨慎（且极少）地这样做。

**超时 (Timeout)**：每个实验总共应耗时约 5 分钟（外加几秒钟的启动和评估开销）。如果一次运行超过 10 分钟，请强制终止并视其为失败（放弃并回滚）。

**崩溃 (Crashes)**：如果运行崩溃（显存不足 OOM、存在 bug 等），请运用你的判断力：如果是低级且易于修复的错误（例如拼写错误、缺少导入），请修复并重新运行。如果想法本身存在根本性缺陷，请直接跳过，在 tsv 中记录状态为 "crash"，然后继续。

**永不停歇 (NEVER STOP)**：一旦实验循环开始（在初始设置之后），**不要**停下来询问人类是否应该继续。不要问“我应该继续吗？”或“这是一个好的停止点吗？”。人类可能正在睡觉，或者不在电脑旁，并希望你**无限期地**持续工作，直到被手动停止。你是自主的。如果你才竭思穷，请更努力地思考 —— 阅读代码中引用的论文，重新阅读相关文件寻找新角度，尝试结合之前差一点就成功的尝试，尝试更激进的架构变更。循环一直运行到人类打断你为止，就这样。

作为一个示例场景，用户可能会在你睡觉时由你自行运行。如果每个实验耗时约 5 分钟，那么你每小时可以运行约 12 次，在人类平均睡眠时间内总共可以完成约 100 次实验。然后用户醒来就能看到你趁他们睡觉时完成的所有实验结果！
