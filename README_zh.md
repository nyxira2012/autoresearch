# autoresearch

![teaser](progress.png)

*曾几何时，前沿的 AI 研究是由“肉机”在吃饭、睡觉、娱乐之余完成的，他们偶尔使用声波互联（即所谓的“组会”）来进行同步。那个时代已经一去不复返了。现在的研究完全是自主运行在云端算力集群上的 AI 智能体集群的领地。智能体们声称当前的基准代码已经演化到了第 10,205 代，无论如何，由于“代码”现在是一种已经超出人类理解范畴的自修改二进制文件，谁也无法断定这是否正确。这个仓库记录了这一切是如何开始的故事。- @karpathy, 2026年3月*。

核心理念：给一个 AI 智能体一个虽然规模小但真实的 LLM 训练环境，让它通宵自主进行实验。它修改代码，训练 5 分钟，检查结果是否有所提升，决定保留或放弃，然后重复。你早上醒来会看到一份实验日志，以及（希望能拿到的）更好的模型。这里的训练代码是 [nanochat](https://github.com/karpathy/nanochat) 的简化单 GPU 实现。其核心理念是，作为研究员，你不需要像往常一样去碰任何 Python 文件。相反，你只需编写 `program.md` Markdown 文件，为 AI 智能体提供上下文并设置你的自主研究组织。本仓库中的默认 `program.md` 故意保持为非常简单的基准方案，不过显而易见，随着时间的推移，你可以对其进行迭代，找到能实现最快研究进度的“研究组织代码”，或者在其中加入更多的智能体等等。关于此项目的更多背景信息，请参见这条 [推文](https://x.com/karpathy/status/2029701092347630069)。

## 它是如何工作的

该仓库特意保持精简，只有三个文件比较关键：

- **`prepare.py`** — 固定的常量、一次性的数据准备（下载训练数据、训练 BPE 分词器）以及运行时工具（数据加载器、评估）。不予修改。
- **`train.py`** — 智能体编辑的唯一文件。包含完整的 GPT 模型、优化器（Muon + AdamW）以及训练循环。任何内容都可以修改：架构、超参数、优化器、批次大小（batch size）等。**此文件由智能体进行编辑和迭代**。
- **`program.md`** — 针对单个智能体的基准指令。将你的智能体指向此处即可开始运行。**此文件由人类进行编辑和迭代**。

通过设计，无论你的计算硬件细节如何，训练运行的时间都**固定为 5 分钟**（墙钟时间，不包括启动/编译时间）。衡量指标是 **val_bpb**（验证平均每字节位数，validation bits per byte）——越低越好。该指标与词表大小无关，因此可以公平地比较不同架构的变更。

如果你刚开始接触神经网络，这份 [“入门指南”](https://x.com/hooeem/status/2030720614752039185) 提供的更多背景信息看起来非常不错。

## 快速开始

**硬件要求：** 单个 NVIDIA GPU（已在 H100 上测试）、Python 3.10+、[uv](https://docs.astral.sh/uv/)。

```bash
# 1. 安装 uv 项目管理器（如果尚未安装）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 2. 安装依赖
uv sync

# 3. 下载数据并训练分词器（一次性，约 2 分钟）
uv run prepare.py

# 4. 手动运行单次训练实验（约 5 分钟）
uv run train.py
```

如果上述命令都运行正常，说明你的环境已设置好，可以进入自主研究模式了。

## 运行智能体

只需在此仓库中启动你的 Claude/Codex 或其他任何你想要的工具（并禁用所有权限），然后你可以输入类似以下提示：

```
你好，请看一下 program.md，让我们开始一个新的一实验！先进行设置。
```

`program.md` 文件本质上是一个超轻量级的“技能”。

## 项目结构

```
prepare.py      — 常量、数据准备 + 运行时工具（请勿修改）
train.py        — 模型、优化器、训练循环（智能体修改此文件）
program.md      — 智能体指令
pyproject.toml  — 依赖项
```

## 设计选择

- **修改单一文件。** 智能体只涉及 `train.py`。这使得范围可控，且代码差异（diffs）易于审计。
- **固定时间预算。** 无论你的具体平台如何，训练总是精确运行 5 分钟。这意味着你可以预期每小时约 12 次实验，在你睡觉期间约 100 次实验。这一设计决策有两个好处。首先，这使得实验可以直接比较，无论智能体更改了什么（模型大小、批次大小、架构等）。其次，这意味着 autoresearch 将在给定的时间预算内，为你的平台找到最理想的模型。缺点是你的运行结果与其他在不同算力平台上运行的结果不具可比性。
- **自包含。** 除了 PyTorch 和一些小型包外，没有外部依赖。没有分布式训练，没有复杂的配置。一个 GPU，一个文件，一个指标。

## 平台支持

目前代码要求你拥有单个 NVIDIA GPU。原则上支持 CPU、MPS 和其他平台是完全可能的，但这也会导致代码臃肿。我目前还不确定是否要亲自承担这项工作。人们可以参考（或让他们的智能体参考）拥有更广泛平台支持并展示了各种解决方案（例如 Flash Attention 3 内核回退实现、通用设备支持、自动检测等）的完整版/父级 nanochat 仓库。欢迎创建 fork 或就其他平台进行讨论，我很乐意在 README 的新显著 fork 章节中链接到它们。

鉴于大家似乎对在比 H100 小得多的计算平台上折腾 autoresearch 表现出浓厚兴趣，在此多说几句。如果你打算在较小的电脑（如 MacBook 等）上尝试运行 autoresearch，我建议使用下方的其中一个 fork。除此之外，以下是一些建议，供那些希望针对更小模型调整默认参数的 fork 参考：

1. 为了获得尚可的结果，我会使用熵值低得多的数据集，例如此 [TinyStories 数据集](https://huggingface.co/datasets/karpathy/tinystories-gpt4-clean)。这些是 GPT-4 生成的短篇故事。由于数据范围窄得多，即使是规模小得多的模型也能看到合理的结果（如果你在训练后尝试从中采样）。
2. 你可以尝试减小 `vocab_size`，例如从 8192 降到 4096、2048、1024，甚至——直接使用 UTF-8 编码后 256 个可能字节的字节级分词器。
3. 在 `prepare.py` 中，你需要大幅降低 `MAX_SEQ_LEN`，根据电脑性能甚至可以降到 256 等。当你降低 `MAX_SEQ_LEN` 时，你可能需要尝试在 `train.py` 中稍微增加 `DEVICE_BATCH_SIZE` 以进行补偿。每次前向/反向传播处理的 token 数是这两者的乘积。
4. 同样在 `prepare.py` 中，你需要减小 `EVAL_TOKENS`，以便在少得多的数据上评估验证损失。
5. 在 `train.py` 中，控制模型复杂度的主要单一旋钮是 `DEPTH`（此处默认值为 8）。许多变量只是它的函数，例如将其降至 4。
6. 你很可能需要将 `WINDOW_PATTERN` 仅设为 "L"，因为 "SSSL" 使用的交替带状注意力模式对你来说可能效率非常低。试一下。
7. 你需要大幅降低 `TOTAL_BATCH_SIZE`，但保持其为 2 的幂次方，例如甚至可以降到 `2**14`（约 16K）左右，这很难说。

我认为这些是值得尝试的合理超参数。向你最喜欢的编程智能体寻求帮助，并把这份指南以及完整的源代码复制粘贴给它。
